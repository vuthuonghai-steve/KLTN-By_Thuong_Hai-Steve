  🎯 Những Hướng Exploration Thú Vị

  1. "Meta-skill" hay "CLI script"?

  Có 2 cách tiếp cận rất khác nhau:

  ┌─────────────────────────────────────────────────────────────────┐
  │  APPROACH A: Meta-Skill (Orchestrator Agent)                   │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   You (in conversation)                                         │
  │        │                                                        │
  │        ▼                                                        │
  │   pipeline-runner (sub-agent) ◄── This IS a skill             │
  │        │                                                        │
  │        ├──▶ skill-1 (sub-agent)                                 │
  │        ├──▶ skill-2 (sub-agent)                                 │
  │        └──▶ skill-3 (sub-agent)                                 │
  │                                                                 │
  │   Pros: Lives in conversation, can ask you questions           │
  │   Cons: Still uses main context for coordination               │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────┐
  │  APPROACH B: Headless CLI (Bash Script)                         │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │   pipeline-runner.sh                                            │
  │        │                                                        │
  │        ├── claude -p "run skill-1" ──▶ skill-1 output         │
  │        │                            (file)                      │
  │        ├── claude -p "run skill-2" ──▶ skill-2 output         │
  │        │                            (file)                      │
  │        └── claude -p "run skill-3" ──▶ skill-3 output         │
  │                                                                 │
  │   Pros: Complete isolation, no context bleed                   │
  │   Cons: No conversational feedback, harder to debug             │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

  Question cho bạn: Bạn muốn pipeline này hoạt động trong conversation (Approach
   A) hay completely detached (Approach B)?

  2. Stateful handoff — Ai quản lý "đũa"?

  Tài liệu đề xuất dùng _queue.json làm source of truth. Nhưng có một nuance:

  ┌─────────────────────────────────────────────────────────────────┐
  │  WHO MANAGES STATE?                                            │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │  Option A: Centralized (Queue File is Boss)                    │
  │  ┌──────────┐    ┌──────────┐    ┌──────────┐                │
  │  │ Runner   │───▶│ Read     │───▶│ Update   │                │
  │  │          │    │ _queue   │    │ _queue   │                │
  │  └──────────┘    └──────────┘    └──────────┘                │
  │       ▲                                              │         │
  │       │                                              │         │
  │       └──────────────────────────────────────────────┘         │
  │                    All state in one file                        │
  │                                                                 │
  │  Option B: Distributed (Each skill manages its own)            │
  │  ┌──────────┐    ┌──────────┐    ┌──────────┐                │
  │  │ skill-1  │───▶│ skill-1  │───▶│ skill-2  │                │
  │  │ status   │    │ writes   │    │ reads    │                │
  │  │ .yaml    │    │ next.yaml │    │ next.yaml│                │
  │  └──────────┘    └──────────┘    └──────────┘                │
  │                                                                 │
  │  Each skill knows "who's next" — less coupling                 │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

  Question: Bạn muốn runner "biết tudo" hay mỗi skill tự biết mình xong thì gọi
  ai?

  3. "One pipeline to rule them all" hay "Many pipelines"?

  Hiện tại bạn có pipeline cho design phase:
  flow → sequence → class → activity → schema → ui-architecture → pencil

  Nhưng sắp tới khi vào Life-3 (implementation), sẽ có pipeline khác:
  payload-collection → hook-generator → api-endpoint → crud-screen

  ┌─────────────────────────────────────────────────────────────────┐
  │  SHOULD PIPELINES BE SINGLETON OR MULTIPLE?                     │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │  .skill-context/                                                │
  │  ├── pipeline.design.yaml    # For Life-2                      │
  │  ├── pipeline.implementation.yaml  # For Life-3                │
  │  └── pipeline.*.yaml        # Extensible                       │
  │                                                                 │
  │  OR just one:                                                   │
  │                                                                 │
  │  .skill-context/                                                │
  │  └── pipeline.yaml                                              │
  │      ├── design_stages: [...]                                   │
  │      └── implementation_stages: [...]                           │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

  🎯 3 Lớp Kiến Trúc Rõ Ràng

  Lớp 1: Configuration (Tĩnh - Do Steve viết)

  pipeline.yaml          ← "Luật chơi" - Steve đĩa ịnh ngh1 lần
  ├── version: "1.0"
  ├── timeout_per_stage: 1800  # 30 phút = 1800s
  └── stages:
      - id: stage_01
        skill: flow-design-analyst
        depends_on: []
        skip_if: null  # hoặc điều kiện
      - id: stage_02
        skill: sequence-design-analyst
        depends_on: [stage_01]
        skip_if: "output/stage_01/.skip"

  Điểm thiết kế hay: skip_if là một file hoặc biểu thức — Runner không cần hiểu
  logic phức tạp, chỉ check file tồn tại.

  ---
  Lớp 2: Runtime State (Động - Runner quản lý)

  _queue.json           ← "Bảng điểm" - Runner cập nhật mỗi tick
  ├── pipeline_id: "pipe-20260228-001"
  ├── started_at: "2026-02-28T10:00:00Z"
  ├── current_stage: "stage_02"
  ├── stages:
  │   ├── stage_01:
  │   │   ├── skill: flow-design-analyst
  │   │   ├── status: COMPLETED
  │   │   ├── completed_at: "2026-02-28T10:05:00Z"
  │   │   ├── output_version: 1
  │   │   └── handoff_note: "Check Payment edge-case..."
  │   │
  │   ├── stage_02:
  │   │   ├── skill: sequence-design-analyst
  │   │   ├── status: IN_PROGRESS
  │   │   ├── started_at: "2026-02-28T10:05:01Z"
  │   │   └── attempt: 1
  │   │
  │   └── stage_03:
  │       ├── skill: class-diagram-analyst
  │       ├── status: BLOCKED_BY: [stage_02]
  │       └── blocked_reason: "waiting for stage_02"
  │
  └── history:  # Timeline of all attempts
      - stage_01:
          - attempt: 1
            started: "10:00:00"
            completed: "10:05:00"
            result: SUCCESS

  Điểm thiết kế hay: history lưu toàn bộ timeline — dễ dàng debug khi có lỗi.

  💎 3 Design Patterns Quan Trọng

  Pattern 1: "Checkpoint Resume"

  ┌─────────────────────────────────────────────────────────────────┐
  │ CHECKPOINT RESUME PATTERN                                      │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │ Nếu pipeline chạy đến stage_03 mà bị interrupt (Ctrl+C, lỗi): │
  │                                                                 │
  │   _queue.json đã lưu:                                          │
  │   stage_01: COMPLETED                                           │
  │   stage_02: COMPLETED                                           │
  │   stage_03: IN_PROGRESS (hoặc FAILED)                          │
  │                                                                 │
  │ Khi Steve chạy lại:                                            │
  │   Pipeline Runner đọc queue → thấy stage_01,02 done            │
  │   → Tự động resume từ stage_03                                 │
  │   → Không chạy lại từ đầu!                                     │
  │                                                                 │
  │ Implementation:                                                  │
  │   if queue.stages[id].status == IN_PROGRESS:                   │
  │       # Resume mode                                             │
  │       if confirm("Resume from stage_03?"):                      │
  │           # Retry stage_03                                       │
  │       else:                                                     │
  │           # Reset stage_03 và chạy lại                          │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

  🤔 Một Số Edge Cases Cần Xử Lý

  1. Circular Dependencies

  # Ai detect?
  # pipeline.yaml
  - id: A depends_on: [C]
  - id: B depends_on: [A]
  - id: C depends_on: [B]

  → Runner phải validate DAG trước khi chạy!

  #!/usr/bin/env python3
  """
  Skill Validation Script Template
  Location: .agent/skills/{skill-name}/scripts/validate.py
  """

  import json
  import sys
  from pathlib import Path
  from typing import Dict, List, Any

  # ============== CONFIG ==============
  SKILL_NAME = "sequence-design-analyst"
  OUTPUT_DIR = Path(".skill-context/results/stage_02/")
  SKILL_DIR = Path(".agent/skills/sequence-design-analyst")

  # ============== LOAD CONTRACT ==============
  def load_contract() -> Dict:
      """Đọc contract từ SKILL.md frontmatter"""
      import yaml
      skill_md = SKILL_DIR / "SKILL.md"
      # Parse YAML frontmatter...
      return {
          "output_contract": [...],
          "quality_gates": {...}
      }

  # ============== STRUCTURAL VALIDATION ==============
  def validate_structure(contract: Dict) -> Dict:
      """Kiểm tra output files tồn tại và đúng format"""
      results = {
          "passed": True,
          "errors": [],
          "warnings": []
      }

      for item in contract["output_contract"]:
          path = OUTPUT_DIR / item["path"]

          if not path.exists():
              results["passed"] = False
              results["errors"].append(f"Missing: {item['path']}")
              continue

          if item["format"] == "json":
              try:
                  json.loads(path.read_text())
              except json.JSONDecodeError as e:
                  results["passed"] = False
                  results["errors"].append(f"Invalid JSON in {item['path']}: 
  {e}")

          # ... more format checks

      return results

  # ============== CONTRACT VALIDATION ==============
  def validate_contract(contract: Dict) -> Dict:
      """Kiểm tra output đáp ứng successor needs"""
      results = {
          "passed": True,
          "warnings": []
      }

      # Load predecessor handoff
      handoff_file = OUTPUT_DIR.parent / "stage_01" / "handoff_note.md"
      if handoff_file.exists():
          # Check if predecessor's edge cases are addressed
          pass

      # Check successor requirements
      for successor_need in contract.get("meets_successor_needs", []):
          # Verify output has what successor needs
          pass

      return results

  # ============== QUALITY GATE VALIDATION ==============
  def validate_quality(contract: Dict) -> Dict:
      """Kiểm tra quality gates"""
      results = {
          "passed": True,
          "errors": [],
          "metrics": {}
      }

      gates = contract.get("quality_gates", {})

      # Example: Count flows
      output_file = OUTPUT_DIR / "sequence_diagrams.md"
      if output_file.exists():
          content = output_file.read_text()

          # Count sequences
          sequences = content.count("sequenceDiagram")
          results["metrics"]["sequence_count"] = sequences

          if "min_sequences" in gates:
              if sequences < gates["min_sequences"]:
                  results["passed"] = False
                  results["errors"].append(
                      f"Only {sequences} sequences, need 
  {gates['min_sequences']}"
                  )

      return results

  # ============== MAIN ==============
  def main():
      print(f"🔍 Running validation for {SKILL_NAME}...")

      contract = load_contract()

      # Run all validations
      struct_result = validate_structure(contract)
      contract_result = validate_contract(contract)
      quality_result = validate_quality(contract)

      # Combine results
      all_passed = (
          struct_result["passed"] and
          contract_result["passed"] and
          quality_result["passed"]
      )

      # Output result
      result = {
          "skill": SKILL_NAME,
          "overall_passed": all_passed,
          "structure": struct_result,
          "contract": contract_result,
          "quality": quality_result,
          "timestamp": "2026-02-28T10:00:00Z"
      }

      # Write to validation report
      report_file = OUTPUT_DIR / "validation_report.json"
      report_file.write_text(json.dumps(result, indent=2))

      if all_passed:
          print("✅ VALIDATION PASSED")
          print(f"   Metrics: {quality_result.get('metrics', {})}")
          sys.exit(0)
      else:
          print("❌ VALIDATION FAILED")
          print(f"   Errors: {struct_result.get('errors', []) + 
  quality_result.get('errors', [])}")
          print(f"   Warnings: {contract_result.get('warnings', [])}")
          sys.exit(1)

  if __name__ == "__main__":
      main()
